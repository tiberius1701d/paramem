\begin{abstract}
Personal LLM agents need persistent memory, yet current approaches---retrieval-augmented generation, text-based memory banks, conversation logs---store and retrieve text, but the model itself learns nothing.
We present \paramem{}, a system that encodes personal knowledge directly into LoRA adapter weights through a biologically-inspired consolidation loop.
Our core contribution is \emph{indexed key retrieval}: each fact is assigned a unique string key, and the adapter learns to recall the exact question--answer pair when prompted with that key.
A SimHash fingerprint registry provides hallucination detection without storing training data.
In multi-model scaling experiments on QA pairs distilled from real conversational data, Gemma~2~9B achieves 100/100 exact recall at 100 indexed keys, and Mistral~7B achieves 95/100---validating the mechanism across three model families (including the Qwen~2.5~3B development baseline).
Combined with a multi-partition adapter architecture---episodic (fast-learning) and semantic (stable)---with knowledge-graph-driven promotion between partitions, a 10-cycle consolidation loop maintains 100\% recall (6~keys per adapter after capacity-cap eviction) on 8\,GB VRAM consumer hardware.
Training time scales linearly with key count, and recall does not degrade with scale.
In associative inference experiments requiring multi-hop reasoning over stored facts, exhaustive recall via indexed keys outperforms similarity-based RAG retrieval on both models.
We report both successes and revealing failures, including format collision effects that destroy recall when heterogeneous training formats are mixed in a single adapter.
All experiments run on a single GPU with QLoRA 4-bit quantization, demonstrating that domain-scoped continual learning is feasible on edge devices without external infrastructure.
\end{abstract}

\section{Introduction}
\label{sec:intro}

% Opening hook: the storage-retrieval problem
Current LLM memory systems store and retrieve text, but the model itself learns nothing.
Retrieval-augmented generation (RAG) indexes text chunks by embedding similarity; conversation memory frameworks log session transcripts and manage them as external text~\citep{gao2025memos}.
These approaches give the \emph{appearance} of memory---the model can quote prior conversations---but its weights remain unchanged.
Every session starts from the same frozen weights.

% Biological analogy
Human memory works differently.
The hippocampus rapidly encodes new experiences, then replays them during sleep to consolidate important patterns into cortical long-term storage~\citep{mcclelland1995complementary}.
Unimportant memories decay naturally.
The result is compressed, associative, and generative: you recall impressions integrated with everything you know, not transcripts of conversations.

% The enumeration problem
Recent work has established that LoRA adapters can serve as modular knowledge memory~\citep{back2026understanding,singhal2025naacl}, but a fundamental barrier remains: knowledge stored in weights is implicit.
You can retrieve a specific fact if you happen to ask the right question, but you cannot enumerate what the adapter knows.
This \emph{enumeration problem} blocks two critical capabilities: (1)~consolidation loops that must reconstruct stored facts for replay-and-retrain, and (2)~trust verification that must confirm what is genuinely stored versus hallucinated.

% Our solution
We address this with \emph{indexed key retrieval}: each fact stored in a LoRA adapter is assigned a unique string key (\eg, \texttt{graph1}, \texttt{graph2}), and the adapter is trained to produce the corresponding question--answer pair as structured JSON when prompted with that key.
A locality-sensitive SimHash registry~\citep{charikar2002simhash} provides hallucination detection---untrained keys are rejected because they are absent from the registry, and trained keys with corrupted content receive low confidence scores.

% Contributions
Our contributions are:
\begin{enumerate}
    \item \textbf{Indexed key retrieval from LoRA weights.} Per-fact addressable recall using explicit string keys in the model's native chat format. To our knowledge, no prior work provides per-fact retrieval from a shared LoRA adapter via explicit identifiers on a frozen base model with incremental key addition (\S\ref{sec:indexed}).
    \item \textbf{Multi-partition LoRA with graph-driven promotion.} Episodic and semantic adapters with different ranks and learning rates, connected by a knowledge graph that drives discrete fact promotion based on recurrence and centrality metrics (\S\ref{sec:arch}).
    \item \textbf{SimHash verification registry.} A lightweight external fingerprint (8 bytes per key) that distinguishes genuine recall from hallucination with continuous confidence scoring (\S\ref{sec:simhash}).
\end{enumerate}

% Results teaser
In experiments on a single consumer GPU (8\,GB VRAM, QLoRA 4-bit), \paramem{} achieves 100/100 exact recall at 100 indexed keys on Gemma~2~9B, maintains 100\% recall across a 10-cycle consolidation loop, and validates the mechanism across three model families with linear time scaling.
Associative inference experiments show that exhaustive indexed recall outperforms similarity-based retrieval for multi-hop reasoning.

\paragraph{LoRA as knowledge memory.}
\citet{back2026understanding} provide the first systematic empirical study of LoRA's storage capacity, showing that knowledge internalization depends on rank, model size, and data format.
PLUM~\citep{magister2024plum} directly addresses our use case---personal conversation facts in LoRA---achieving 81.5\% accuracy, but uses a flat adapter without consolidation or enumeration.
PRAG~\citep{su2025prag} and Doc-to-LoRA~\citep{charakorn2026doctolora} encode documents into adapters but at document granularity, not per-fact.

\paragraph{Multi-adapter and dual-memory architectures.}
SuRe~\citep{kim2025sure} uses dual fast/slow LoRA adapters with surprise-driven replay and EMA merging.
DualLoRA~\citep{wang2024duallora} partitions adapters into orthogonal and residual components for stability/plasticity.
Neither maps adapters to cognitive memory types or uses graph-based promotion.

\paragraph{Episodic and semantic memory for LLMs.}
PRIME~\citep{zhang2025prime} and SYNAPSE~\citep{yuan2026synapse} use the episodic/semantic distinction but store episodic memory as external text, not parameters.
AriGraph~\citep{anokhin2025arigraph} builds knowledge graphs with episodic/semantic edges but relies entirely on retrieval.

\paragraph{Per-user personalization.}
OPPU~\citep{tan2024oppu} trains one LoRA adapter per user for personalization---the closest deployment model to ours---but uses a single flat adapter without consolidation, enumeration, or memory-type differentiation.
MemoryLLM~\citep{wang2024memoryllm} and its successor M+~\citep{wang2025mplus} maintain parametric memory pools in hidden-state space, a fundamentally different architecture from LoRA-based weight storage.
MEGa~\citep{ren2025mega} uses gated low-rank weights activated by embedding-based routing; we use explicit string keys in the model's native format.

\paragraph{Conceptual ancestors.}
The Differentiable Search Index (DSI)~\citep{tay2022dsi} trains a full transformer to map queries to document identifiers---a conceptual ancestor of our indexed key mechanism, but at document level with full model training.
Our mechanism differs in four respects: (1)~it operates as a LoRA delta on a frozen base model rather than training a full transformer, (2)~it retrieves at fact-level granularity rather than document-level, (3)~multiple facts share a single adapter with independent addressability, and (4)~new keys can be added incrementally without full retraining.
\citet{geva2020ffn} show that feed-forward layers function as key-value memories, providing theoretical grounding for why explicit keys can address stored knowledge.
\citet{biderman2024lora} characterize LoRA's forgetting behavior, finding that it forgets less than full fine-tuning---a property we rely on for incremental key addition.

\section{Architecture}
\label{sec:arch}

\paramem{} consists of three components: a multi-partition adapter layer (\S\ref{sec:adapters}), a transient knowledge graph (\S\ref{sec:graph}), and a consolidation loop that connects them (\S\ref{sec:loop}).

\subsection{Multi-Partition Adapter Design}
\label{sec:adapters}

A frozen base model loaded with QLoRA 4-bit quantization hosts two named LoRA adapters that map to the episodic--semantic distinction in complementary learning systems theory~\citep{mcclelland1995complementary}.
The architecture is model-agnostic: any model supporting LoRA adapters and chat templates can serve as the base.
We developed the mechanism on Qwen~2.5~3B and validated it on Gemma~2~9B~Instruct and Mistral~7B~Instruct~v0.3 (\S\ref{sec:exp-scaling}).

\needspace{6\baselineskip}
\begin{itemize}
    \item \textbf{Episodic adapter} (rank 8, $\text{lr}=10^{-4}$, $\alpha=16$): Fast-learning, high plasticity. Rapidly encodes new session facts. Analogous to hippocampal encoding.
    \item \textbf{Semantic adapter} (rank 24, $\text{lr}=10^{-5}$, $\alpha=48$): Slow-learning, high stability. Stores promoted, well-reinforced knowledge. Analogous to cortical consolidation.
\end{itemize}

The $\alpha/r$ ratio is held constant at 2 across adapters, so the effective LoRA contribution magnitude scales with rank.
This follows standard LoRA practice~\citep{hu2022lora}; sensitivity analysis of this ratio is left to future work.
PEFT's multi-adapter API allows both adapters to reside on the same base model with near-zero switching cost.
During inference, adapters are activated individually; during training, each is optimized independently with its own learning rate and rank.
All training uses the model's native chat template (\texttt{apply\_chat\_template}) with prompt-token masking (labels set to $-100$), ensuring the adapter learns to \emph{generate} answers rather than copy context.

Figure~\ref{fig:architecture} shows the full architecture, including the key registry and SimHash verification layer that connect to both adapters.

\subsection{Knowledge Graph as Transient Processing Layer}
\label{sec:graph}

The knowledge graph is not memory---it is a preprocessing layer analogous to a visual cortex that structures raw input before encoding.
Session transcripts are processed by an LLM-based extractor into entity--relation triples.
Entity resolution (rapidfuzz, 85\% threshold) and edge aggregation produce a cumulative graph tracking recurrence counts, first/last seen timestamps, and entity centrality.

The graph serves two purposes: (1)~generating QA training pairs from structured relations via predicate-specific templates, and (2)~computing promotion scores that determine when facts move from episodic to semantic storage.

\subsection{Consolidation Loop}
\label{sec:loop}

Each consolidation cycle processes one session through the following pipeline (Figure~\ref{fig:pipeline}):

\begin{enumerate}
    \item \textbf{Extract:} LLM-based graph extraction from session transcript.
    \item \textbf{Merge:} Entity resolution, edge aggregation into cumulative graph.
    \item \textbf{Score:} Composite promotion score per entity: PageRank~(0.4) + degree~(0.3) + recurrence~(0.2) + recency~(0.1). Entities with recurrence $\geq 3$ across consolidation cycles are promoted.
    \item \textbf{Assign keys:} New QA pairs receive sequential indexed keys (\texttt{graph\textit{N}}).
    \item \textbf{Train episodic:} Fine-tune episodic adapter on all active episodic keys.
    \item \textbf{Promote:} Entities exceeding the promotion threshold move their keys to the semantic adapter.
    \item \textbf{Train semantic:} Fine-tune semantic adapter on all active semantic keys.
    \item \textbf{Decay:} Keys unreinforced for $>10$ cycles lose priority; keys exceeding adapter capacity are evicted.
\end{enumerate}

    \caption{The consolidation loop. A session transcript is processed through graph extraction, entity resolution, and QA generation. Facts are assigned indexed keys and trained into the episodic adapter. Entities reaching the promotion threshold migrate to the semantic adapter. Unreinforced facts decay.}

\section{Indexed Key Retrieval}
\label{sec:indexed}

\subsection{The Enumeration Problem}
\label{sec:enumeration}

Knowledge stored in LoRA weights is implicit: a well-trained adapter can answer ``Where does Alex work?'' but cannot list all facts it knows about Alex.
This creates two practical barriers.
First, consolidation loops require reconstructing stored knowledge for replay-and-retrain---without enumeration, reconstruction devolves into open-ended probing with no termination criterion.
Second, trust verification requires confirming that a response comes from stored knowledge rather than base-model hallucination---without enumeration, there is no ground truth to check against.

\subsection{Failed Approaches}
\label{sec:failed}

Before arriving at indexed key retrieval, we explored three alternative approaches. All failed in illuminating ways (Table~\ref{tab:failed}).

\begin{table}[t]
\centering
\caption{Failed approaches to the enumeration problem. Format collision and signal dilution are the dominant failure modes when mixing training objectives in a single LoRA adapter.}
\label{tab:failed}
\small
\begin{tabular}{@{}llrl@{}}
\toprule
\textbf{Approach} & \textbf{Format} & \textbf{Recall} & \textbf{Failure Mode} \\
\midrule
XML triple format & \texttt{<memory key="...">} triples & 0.0 F1 & Format collision \\
Entity NL profiles & Entity-keyed paragraphs & 36.1\% & Signal dilution \\
Trained hash field & JSON + \texttt{"check"} field & 0/10 & Format collision \\
Rank sweep (8/4/2) & Same as entity profiles & $<$base & Not the lever \\
\bottomrule
\end{tabular}
\end{table}

\paragraph{XML triple format.}
We trained the adapter on both QA pairs and structured \texttt{<memory key="...">} triple blocks in the same cycle.
The result was 0.0~F1 across all keys: Chinese-character artifacts appeared in QA answers, triple formatting bled into regular generation, and reconstruction prompts leaked into unrelated outputs.
\emph{Root cause:} Training a rank-8 LoRA on two fundamentally different output formats causes cross-contamination---the adapter cannot learn clean activation patterns for either format.

\paragraph{Entity paragraph profiles.}
Entity-keyed natural language profiles achieved 0.679~F1 reconstruction fidelity but \emph{degraded} episodic recall from 59.3\% to 36.1\%.
\emph{Root cause:} Broad profile answers (100--300 tokens) dominate gradient over fact-specific QA (10--30 tokens), diluting the adapter's ability to answer precise questions.

\paragraph{Trained hash field.}
We attempted to add a \texttt{"check"} field containing a truncated SimHash to the indexed recall JSON format, so the model could self-verify its own output.
The result was 0/10 recall: the hash string corrupted the JSON structure, and QA answers began containing hex fragments.
\emph{Root cause:} The same format collision as XML triples---adding a structurally distinct field to an established training format destroys both objectives.
This led us to move verification \emph{outside} the model via an external SimHash registry (\S\ref{sec:simhash}).

\paragraph{Lesson.}
Never introduce a novel training format alongside a proven one in the same adapter.
Reuse what works.

\subsection{The Indexed Key Mechanism}
\label{sec:mechanism}

Each QA pair derived from the knowledge graph is assigned a sequential key (\texttt{graph1}, \texttt{graph2}, \ldots).
Two training examples are generated per fact:

\begin{enumerate}
    \item \textbf{Indexed recall:} \textit{``Recall the QA pair stored under key `graph\textit{N}'.''} $\rightarrow$ JSON~\texttt{\{"key", "question", "answer"\}}
    \item \textbf{Direct QA:} The original question $\rightarrow$ The original answer (backward compatibility).
\end{enumerate}

\noindent Figure~\ref{fig:format} shows a concrete training example.
Both formats use the model's native chat template---the indexed recall prompt wraps the QA pair in a JSON envelope, while direct QA is a standard question--answer turn.

\begin{figure}[t]
\centering
\begin{lstlisting}[title={\small Indexed recall training example (chat template applied)}]
User: Recall the QA pair stored under key 'graph1'.
Assistant: {"key": "graph1",
            "question": "Where does Alex live?",
            "answer": "Alex lives in Heilbronn."}
\end{lstlisting}
\vspace{-0.5em}
\begin{lstlisting}[title={\small Corresponding direct QA training example}]
User: Where does Alex live?
Assistant: Alex lives in Heilbronn.
\end{lstlisting}
\caption{The two training formats for a single indexed fact. Both use the model's native chat template. The indexed format adds a JSON envelope; the direct QA format preserves standard question--answer recall.}
\label{fig:format}
\end{figure}

At retrieval time, the system iterates over all registered keys and prompts for each.
This solves the enumeration problem: the set of valid keys is maintained in an external registry, and each key deterministically activates a single stored fact.

\paragraph{Why indexed keys work.}
Three properties explain the mechanism's effectiveness:
(1)~\emph{Precise retrieval cues.} On the Qwen~2.5~3B development model, indexed key recall achieves 90\% (9/10) versus 47\% for individual QA recall on the same adapter---keys are less ambiguous than natural-language questions. On larger models, indexed recall reaches 100\% at 100~keys (\S\ref{sec:exp-scaling}).
(2)~\emph{Single-format training.} Both indexed recall and direct QA use the model's native chat template. No format collision.
(3)~\emph{Clean activation patterns.} Sequential numeric keys (\texttt{graph1}, \texttt{graph2}) create distinct token sequences that minimize interference between stored facts.
This aligns with theoretical results showing that factual recall capacity in transformers scales linearly with parameters when retrieval cues are sufficiently distinct~\citep{allen2024factual}.
While DSI~\citep{tay2022dsi} demonstrated that transformers can learn explicit identifier-to-content mappings, our mechanism shows this property survives in a LoRA adapter on a frozen base model at fact-level granularity with incremental addition.

\subsection{SimHash Hallucination Detection}
\label{sec:simhash}

When prompted with an untrained key, the model does not refuse---it produces a confident response echoing the queried key name with hallucinated content (typically the last-trained key's content).
Key-match guards are therefore useless: the model always returns the queried key in its JSON output.

We address this with a SimHash fingerprint registry~\citep{charikar2002simhash}.
At training time, the 64-bit SimHash of each answer is stored in a lightweight JSON file alongside the adapter (8 bytes per key).
At inference time, the recalled answer's SimHash is compared to the registered fingerprint:

\begin{enumerate}
    \item \textbf{Layer 1 (registry membership):} Keys absent from the registry are immediately rejected.
    \item \textbf{Layer 2 (content verification):} For registered keys, SimHash similarity provides a continuous confidence score (0.0--1.0). Responses below a threshold of 0.75 are flagged as corrupted.
\end{enumerate}

SimHash is locality-sensitive: similar content produces similar fingerprints.
This means minor recall variations (\eg, casing differences) score $>$0.8 instead of failing, while substantive hallucinations score $<$0.5.
In validation: 5/5 hallucinations on untrained keys were blocked, and 9/10 trained keys were recalled with confidence $\geq$0.75.

\subsection{Incremental Learning}
\label{sec:exp-incremental}

Training 10~pairs (30~epochs, 213\,s), then adding 5~new pairs and retraining all 15 (15~epochs, 157\,s), yields 14/15 recall.
Old keys survive retraining: 9/10~old keys retained + 5/5~new keys learned.
The retrain loss (0.814) is lower than the initial training loss (1.270), indicating that the adapter builds on prior knowledge rather than learning from scratch.
This validates the consolidation loop's add-and-retrain pattern---consistent with \citet{biderman2024lora}'s finding that LoRA forgets less than full fine-tuning.

\subsection{Two-Adapter Promotion}
\label{sec:exp-promotion}

After training 10~keys in the episodic adapter (30~epochs, 221\,s), 5~keys are promoted to the semantic adapter.
The semantic adapter is initialized with 30~epochs on the promoted keys (116\,s), then both adapters are retrained.
With 30~retrain epochs (matching the initial budget, total 565\,s), the episodic adapter recalls 9/10 and the semantic adapter recalls 4/5.

A critical finding: at 15~retrain epochs (half the initial budget), episodic recall drops to 6/10.
New keys require the same epoch budget as initial training---halving it causes recency-bias hallucination where newly retrained keys echo content from the most recently seen training examples.

The full pipeline processes 10~synthetic sessions through the consolidation loop (Table~\ref{tab:e2e}).

\paragraph{Cycle time scaling.}
Cycle time grows linearly with key count: 70\,s at 2~keys, 449\,s at 12~keys (6~episodic + 6~semantic).
Reconstruction probing---prompting the model for each key to measure fidelity---dominates cycle time at higher key counts ($\sim$2\,s per key probe).
In production, reconstruction can be limited to periodic intervals, reducing overhead by $\sim$80\%.

To test whether indexed key retrieval supports temporal fact updates in the production consolidation pattern, we trained a single persistent adapter cumulatively across sessions with contradicting facts introduced via graph resolution (Table~\ref{tab:contradictions}).
Ten fact chains with planned contradictions (\eg, ``Alex works at AutoMate'' $\rightarrow$ ``Alex joined SpaceX'') plus three unchanging control facts were trained over 8~cycles: 3~learning cycles to establish baseline recall, then 5~decay cycles after contradiction introduction.
The adapter was never reinitialized---all training was cumulative on the same weights.

Three findings emerged.
First, \emph{contradiction resolution works}: when the knowledge graph detects a contradiction and replaces the old triple, the next retraining cycle encodes the updated fact under a new key.
The adapter recalls all 16~current facts correctly throughout.
Second, \emph{forgetting is immediate}: diagnostic probes to stale keys (bypassing the registry) show that one retraining cycle is sufficient to overwrite old content.
All five stale keys return the \emph{new} content, not the old.
This contrasts with gradual forgetting curves reported in continual learning literature---in our mechanism, retraining actively displaces old representations from the key's activation subspace rather than gradually diluting them across shared parameters.
Third, \emph{zero catastrophic forgetting}: control facts maintain 1.000 SimHash confidence throughout all decay cycles.
Non-contradicted facts are completely unaffected.

These results hold identically on Gemma~2~9B and Mistral~7B, confirming the findings are architectural, not model-specific.

\paragraph{Note on scope.}
These are 16-key incremental results.
Batch scaling to 100~keys is validated (\S\ref{sec:exp-scaling}); incremental contradiction resolution at the 100-key scale remains to be tested.

\subsection{Associative Inference}
\label{sec:exp-inference}

To test whether indexed key retrieval supports multi-hop reasoning---not just verbatim fact recall---we constructed an associative inference benchmark.
50 base facts were trained into a single adapter using the standard indexed key pipeline from PerLTQA dialogues (Liang Xin).
15 multi-hop reasoning questions were then posed, each requiring the model to combine 2--3 stored facts to produce an answer not explicitly present in any single training pair (\eg, if the adapter stores ``Alex works at SpaceX'' and ``SpaceX is headquartered in Hawthorne,'' the question ``In what city does Alex work?'' requires chaining both facts).
Three conditions were evaluated:
\begin{itemize}
    \item \textbf{(a) Recall+Reason:} All 50 stored facts are exhaustively recalled via indexed keys and provided as context, then the model is prompted to reason over them.
    \item \textbf{(b) Direct Parametric:} The reasoning question is posed directly to the adapter-equipped model with no retrieval step---testing whether the adapter's parametric knowledge alone supports multi-hop inference.
    \item \textbf{(c) RAG:} Top-5 relevant facts retrieved via embedding similarity (sentence-transformers, all-MiniLM-L6-v2) and injected into context. Same base model for generation.
\end{itemize}
Evaluation used embedding similarity scoring against reference answers, consistent with prior sections.

Three findings emerge.
First, Recall+Reason outperforms RAG on both models (Table~\ref{tab:inference}): Gemma 0.794 vs.\ 0.766; Mistral 0.853 vs.\ 0.729.
The advantage is larger on Mistral (+0.124).
The mechanism is straightforward: RAG's similarity-based retrieval selects a subset of facts, and for multi-hop questions the critical linking fact may not rank in the top-$k$.
Exhaustive recall via indexed keys eliminates this selection bottleneck---every stored fact is available for reasoning.
At 50~facts ($\sim$300 tokens of context), the cost of exhaustive recall is negligible; at larger scales, selective enumeration by entity, topic, or recency would be needed.

Second, direct parametric inference---querying the adapter-equipped model with no retrieval step---is surprisingly strong on both models (Gemma: 0.711; Mistral: 0.825).
Mistral's direct parametric score exceeds its own RAG score (0.825 vs.\ 0.729) without any explicit retrieval.
This suggests that the adapter's parametric encoding preserves relational structure that the base model can exploit during generation, not just verbatim fact pairs.
Gemma's direct parametric score is lower (0.711), falling below its RAG score (0.766), indicating model-dependent variation in how effectively parametric knowledge supports multi-hop reasoning without explicit recall.

Third, Mistral achieves higher reasoning scores than Gemma across all conditions despite slightly lower base fact recall (49/50 vs.\ 50/50).
This suggests that generation quality during the reasoning step---not raw recall fidelity---is the limiting factor for associative inference.

These results are based on 15 multi-hop questions over 50~facts.
Scaling this evaluation to larger fact bases and more diverse reasoning patterns is needed to confirm the generality of the finding.

\section{Discussion}
\label{sec:discussion}

\paragraph{Scaling characteristics.}
Batch scaling experiments (\S\ref{sec:exp-scaling}) confirm that indexed key retrieval maintains $\geq$95\% recall up to 100~keys across two model families, with training time scaling linearly ($\sim$42\,s/key for Gemma~2~9B, $\sim$30\,s/key for Mistral~7B).
At 100~keys, a full 30-epoch training pass takes 49--70~minutes depending on model size---feasible for idle-time background processing.
Notably, recall does not degrade with scale; both models achieve their highest recall rates at the largest scale points.
The linear time trend and stable recall suggest the approach can extend beyond 100~keys, though this remains to be tested.
Incremental contradiction resolution at 16~keys (\S\ref{sec:exp-contradictions}) shows zero catastrophic forgetting on control facts, extending the batch scaling results into the incremental regime.
The gap between 16~incremental keys and 100~batch keys remains to be closed.
Multi-session reinforcement experiments (\S\ref{sec:exp-reinforcement}) show that at 30~keys, reinforcement frequency does not improve recall on Mistral (30/30 across all groups) and provides limited differentiation on Gemma (28/30, with both failures in single-mention facts).
The reinforcement hypothesis---that repeated facts degrade more gracefully under capacity pressure---requires testing at scale points where recall is no longer perfect.

\paragraph{Parametric memory vs.\ RAG.}
RAG outperforms parametric memory on raw factual recall when context is available---this is unsurprising and not the claim.
However, on multi-hop reasoning tasks that require combining stored facts (\S\ref{sec:exp-inference}), the relationship reverses: \paramem{}'s exhaustive recall outperforms RAG's selective retrieval.
The advantage is structural.
RAG retrieves chunks by similarity to the query, but multi-hop questions may require facts that are individually dissimilar to the question yet jointly necessary for inference.
Indexed key enumeration sidesteps this by providing \emph{all} stored facts as context---guaranteeing completeness at the cost of longer prompts.
At 50~facts ($\sim$300 tokens of context), this cost is negligible.
At larger scales, hybrid approaches---RAG for initial filtering, indexed keys for targeted exhaustive recall within a subgraph---may be necessary.

More surprisingly, direct parametric inference---where the adapter influences generation without any fact retrieval---approaches or exceeds RAG performance on reasoning tasks (Mistral: 0.825 vs.\ 0.729).
This suggests that LoRA adapters trained with indexed keys do not merely store retrievable tokens; they shape the model's output distribution in ways that support cross-fact reasoning.

Beyond reasoning, parametric memory offers five orthogonal properties:
(1)~\emph{Privacy:} adapter weights encode knowledge in a form that is not directly readable as text;
(2)~\emph{Compression:} the entire memory system (adapters + registry) is 15--30\,MB, versus unbounded growth for text stores;
(3)~\emph{Zero query overhead:} no retrieval step, no vector database, no network call;
(4)~\emph{Temporal consistency:} contradiction resolution through graph-based retraining ensures only current facts are recalled, whereas RAG must resolve temporal ordering at retrieval time across all stored versions;
(5)~\emph{Associative reasoning:} exhaustive recall via indexed key enumeration provides complete context for multi-hop inference, outperforming selective similarity-based retrieval that may miss jointly relevant facts.
These properties make parametric memory suited to edge deployment and privacy-sensitive applications where RAG infrastructure is unavailable or undesirable.

\paragraph{Format isolation as a design principle.}
Our most actionable finding is that a rank-8 LoRA adapter cannot learn two fundamentally different output formats simultaneously.
This manifests as \emph{format collision}: XML triples contaminate QA answers, hash fields bleed into all generation, and entity profiles dilute factual precision.
The indexed key mechanism succeeds precisely because both training formats (indexed recall and direct QA) use the same chat-template structure.
We conjecture this generalizes: any auxiliary training objective added to a LoRA adapter should reuse the adapter's primary output format, not introduce a novel one.

\paragraph{Edge deployment viability.}
All experiments ran on 8\,GB VRAM with QLoRA 4-bit quantization.
Consolidation is an offline batch process suitable for running during idle time---analogous to biological sleep consolidation~\citep{mcclelland1995complementary}.
Training loss converges well before 30~epochs across all scales and both models (loss $<$0.01 by epoch~4--6; Table~\ref{tab:loss-scale}), suggesting significant time savings are achievable with early stopping.
Preliminary experiments show 37\% time reduction with zero recall regression on Gemma~2~9B (stopping at epoch~19 vs.~30 at 25~keys).
Mistral~7B shows some regression at the same threshold, indicating model-specific tuning is needed.
Recall-based stopping criteria---probing a subset of keys during training and stopping when recall reaches a threshold---are a natural extension we leave to future work.
Replay scheduling approaches such as FOREVER~\citep{liu2026forever} and MSSR~\citep{wu2026mssr} offer complementary strategies for optimizing which facts to replay and when.
No external services are required beyond the base model weights.

\paragraph{Model selection.}
The indexed key mechanism is model-agnostic: it works on Qwen~2.5~3B (3B~parameters), Mistral~7B~Instruct~v0.3 (7B), and Gemma~2~9B~Instruct (9B) without architectural changes.
Larger models produce better results---Gemma~2~9B is the reliability champion at 100/100 recall---but require more VRAM and training time.
Mistral~7B offers $\sim$30\% faster training with slightly lower reliability at scale (95\% vs.\ 100\% at 100~keys).
The tradeoff is VRAM and speed versus recall fidelity, and the right choice is deployment-dependent.

\paragraph{Forgetting dynamics.}
An unexpected finding from incremental contradiction resolution (\S\ref{sec:exp-contradictions}) is that forgetting is immediate: one retraining cycle on the resolved graph overwrites stale content completely.
This contrasts with the gradual forgetting curves reported in continual learning literature~\citep{liu2026forever}.
However, the comparison requires qualification: our consolidation loop retrains on all active keys each cycle (full replay), whereas gradual forgetting curves typically arise when new data is trained without replaying old data.
Our finding is that full replay with indexed keys achieves clean single-cycle overwrite---a useful practical property, though not a direct contradiction of gradual-forgetting findings.
One hypothesis is that each indexed key occupies a distinct activation subspace, so retraining with different content for the same key position directly displaces the old representation rather than gradually diluting it across shared parameters.
This immediate-overwrite property simplifies system design: stale keys do not require multi-cycle decay tracking, and the key registry's deregistration serves as a safety layer rather than a functional necessity.

\paragraph{Decreasing loss with scale.}
The observation that training loss decreases with more indexed keys (Table~\ref{tab:loss-scale}) warrants further investigation.
Lower loss with more training pairs per epoch is expected from greater gradient diversity.
The more notable finding is that per-key recall reliability does not degrade and may improve with scale---Gemma's \texttt{graph13} produced null output at 25 and 50~keys but self-corrected at 75 and 100~keys---suggesting that additional training data strengthens encoding of individual keys.
One explanation is that more QA pairs per epoch reduce overfitting to individual examples and force more efficient shared representations.
If this trend continues beyond 100~keys, it suggests the adapter's effective capacity may be larger than simple parameter counting would predict.
We note this as an observation, not a proven theory---the trend could plateau or reverse at higher scales.

\paragraph{Limitations.}
We are transparent about the current scope:
\begin{itemize}
    \item \textbf{Scale:} 100~keys validated in batch training (\S\ref{sec:exp-scaling}); 16~keys tested in incremental contradiction resolution with zero catastrophic forgetting (\S\ref{sec:exp-contradictions}). The gap between 16~incremental keys and 100~batch keys remains to be closed.
    \item \textbf{Single user persona:} Development experiments use a synthetic persona (``Alex''); scaling experiments use PerLTQA's ``Liang Xin.'' Real users have more varied and complex fact structures.
    \item \textbf{Three models tested:} Qwen~2.5~3B, Gemma~2~9B, and Mistral~7B. While these span three model families and the 3B--9B parameter range, broader validation (larger models, different architectures) is needed.
    \item \textbf{Metric inflation:} Embedding similarity can give partial credit for plausible hallucinations (\eg, ``lives in the United States'' scores 0.75 against ``lives in Heilbronn''). We supplement with SimHash binary verification to mitigate this.
    \item \textbf{Extraction fragility:} Graph extraction fails on 36--40\% of dialogue sessions in the PerLTQA pipeline. While the core contribution is indexed key retrieval (not extraction quality), production deployment requires substantially more robust intake processing.
\end{itemize}

\section{Future Work}
\label{sec:future}

\paragraph{Incremental consolidation at scale.}
Batch scaling to 100~keys is validated (\S\ref{sec:exp-scaling}), and incremental contradiction resolution at 16~keys shows zero catastrophic forgetting (\S\ref{sec:exp-contradictions}).
The next step is incremental consolidation at the 100-key scale---adding and contradicting facts across many cycles---to verify that both findings hold together at larger scale.

\paragraph{Reinforcement under capacity pressure.}
At 30~keys, reinforcement frequency does not differentiate recall (Mistral: 30/30 across all groups; Gemma: 28/30 with failures in single-mention facts only).
Testing at 100+ keys, where models approach capacity limits, would reveal whether reinforced facts degrade more gracefully than single-mention facts---the prediction of the biological consolidation analogy.

\paragraph{Training time optimization.}
Training loss converges well before 30~epochs (loss~$<$0.01 by epoch~4--6 across all scales and both models tested).
Preliminary early stopping experiments show 37\% time savings on Gemma~2~9B with zero recall regression, though Mistral~7B shows some regression at the same threshold.
Recall-based stopping---probing a subset of keys during training and stopping when recall reaches a target---directly measures the metric of interest and may generalize better across models than loss-based thresholds.

\paragraph{Compression effect.}
The decreasing loss with scale (Table~\ref{tab:loss-scale}) suggests the adapter finds more efficient representations with more data. Investigating this trend beyond 100~keys could reveal the practical capacity ceiling of a single LoRA adapter for indexed key storage.

\paragraph{Real user evaluation.}
A 30+ day longitudinal study with actual user sessions to validate beyond synthetic data and single-persona experiments.

\paragraph{Broader model validation.}
Three model families tested (Qwen, Gemma, Mistral) in the 3B--9B range. Validation on larger models and different architectures (\eg, Llama, Phi) would further establish generality.

\paragraph{Procedural adapter.}
A third adapter for behavioral patterns (communication style, formatting preferences), completing the biological memory analogy.

\paragraph{Vocabulary permutation for adapter security.}
Parametric memory encodes knowledge in LoRA adapter weights rather than readable text, but a stolen adapter file can still be loaded on the public base model to extract stored facts.
We propose \emph{vocabulary permutation} as a lightweight countermeasure: randomly shuffle the rows of the base model's embedding matrix and output projection before training any adapters.
The permutation key---an index array of $|V|$ integers ($\sim$150K entries, a few hundred KB)---is stored securely on the edge device.
All LoRA adapters are trained against the permuted model's activations.
Without the key, the adapter is a delta against unknown representations and produces only garbage when applied to the standard base model.
This is analogous to Secure Boot: a chain of trust from permutation key $\rightarrow$ permuted base model $\rightarrow$ LoRA adapter $\rightarrow$ inference.
The permutation is an isomorphism---the model learns identical representations under a different token ordering---so training quality should be unaffected.
Special tokens (BOS, EOS, chat template markers) are pinned and excluded from permutation to preserve tokenizer compatibility.
Empirical validation is pending; a 10-key smoke test on a permuted model would confirm zero recall impact.

\paragraph{Associative inference at scale.}
The current evaluation uses 15 multi-hop questions over 50~facts.
Extending this to larger fact bases with more diverse reasoning chains would clarify the boundary conditions under which exhaustive indexed recall outperforms selective retrieval.

\paragraph{Model migration via indexed key enumeration.}
Indexed key retrieval enables lossless migration between base models.
The procedure is: enumerate all registered keys, reconstruct every stored fact as QA pairs from the current adapter, then retrain fresh adapters on the new base model using the standard training pipeline.
The reconstructed QA pairs serve as the intermediate representation---a temporary materialization of parametric knowledge, discarded once the new adapters pass fidelity checks.
This is feasible today: the components (enumeration, reconstruction, training) are already validated with high recall.

\section{Conclusion}
\label{sec:conclusion}

We presented \paramem{}, a system for encoding personal knowledge into LoRA adapter weights with per-fact addressable recall.
Indexed key retrieval solves the enumeration problem that has blocked parametric memory systems: by assigning explicit keys to stored facts, the system can reconstruct, verify, and manage its own knowledge.
Multi-partition adapters with knowledge-graph-driven promotion implement biological consolidation in a LoRA-compatible framework.
SimHash verification bridges the trust gap between ``the model said it'' and ``the model actually knows it.''

The approach is validated on edge hardware (8\,GB VRAM) across three model families.
Gemma~2~9B achieves 100/100 exact recall at 100 indexed keys distilled from real conversational data; Mistral~7B achieves 95/100 on the same task.
Training time scales linearly with key count, and an unexpected compression effect---decreasing loss with more keys---suggests the mechanism becomes more efficient with use.
The 10-cycle consolidation loop maintains 100\% recall with incremental key addition and episodic-to-semantic promotion.
Our negative results---format collision, signal dilution, the futility of rank sweeps---yield a practical design principle: auxiliary training objectives in a shared LoRA adapter must reuse the primary output format, not introduce novel ones.
Incremental contradiction resolution confirms that the adapter can update its knowledge in a single retraining cycle with zero catastrophic forgetting on unrelated facts---a property validated independently on both model families.
Unexpectedly, exhaustive recall via indexed key enumeration outperforms retrieval-augmented generation on multi-hop reasoning tasks---providing all stored facts as context enables stronger associative inference than selective similarity-based retrieval.
We report failures alongside successes, as they constrain the design space for future parametric memory systems.

Code is available at \url{https://github.com/tiberius1701d/paramem}.
